## 프로파일러는 어떤 경우에 유용할까?
프로파일링 도구가 도움이 되는 상황을 세가지정도 꼽아보면 이렇다.

- 비정상적인 리소스 사용량 식별
- 코드의 어느 부분이 실행되는지 찾기
- 앱의 실행 속도가 저하되는 문제 파악

### 비정상적인 리소스 사용량 식별
프로파일러는 대게 CPU와 메모리를 어떻게 쓰는지 파악하는 용도로 쓰인다.

- 스레드 관련 문제
    - 동기화가 결여되어 있거나 제대로 발생되지않는 동시성문제

- 메모리 누수
    - 불필요한 데이터를 메모리에서 비우지 못하여 앱 실행 속도가 느려지고 완전히 멈추는것

모바일 앱에서는 CPU 리소스 사용이 곧 배터리 소모로 이어진다. 이러한 잠재적인 문제를 발견한 후에는 스레드 덤프를 수집해
<br>조사할 수 있다. 근본 원인은 스레드의 동기화와 결함이다.

JVM은 메모리에서 불필요한 데이터를 알아서 비우는 메커니즘, 즉 가비지 컬렉터를 제공하지만 불필요한 데이터를 삭제하는 일은 개발자의 몫이다.
<br>코드를 짜면 GC는 그 객체를 더이상 사용하지 않는다는 사실을 알 도리가 없기 때문에 메모리에서 절대로 수거되지않는다.
<br>이런 상황을 **메모리 누수**라고 그런다.

### 실행되는 코드

프로파일러가 있으면, 코드를 직접 들여다보지 않아도 어느 코드가 백그라운드에서 실행중인지 쉽게 찾을 수 있다.<br>
이런 기능을 **샘플링**이라고 한다. 코드가 너무 복잡해서 무엇이 호출되는지 파악하기 어려울 때 유용하다.

**앱 실행 속도가 느려지는 원인을 파악**
<br>무엇때문에 이렇게 오래걸릴까? 를 의문이 들때가 많다. 개발자는 본능적으로 I/O를 가장 먼저 의심한다.<br>
웹 서비스 호출, DB연결, 데이터를 파일로 저장하는 등의 작업은 흔히 앱에서 레이턴시를 일으키는 사례다.
<br>그러나 I/O만 속도 저하를 일으키는 주범은 아니다. 코드 베이스를 완전히 머릿속에 집어넣지 않는 한<br>
도구없이 문제를 파악하는것은 정말 어렵다.

> 프로파일러는 코드를 가로채서 각 코드가 소비하는 리소스를 계산하는 마법을 부린다.
