## 1.1 앱을 더 쉽게 이해하는 법
- 개발자 대부분이 디버깅이라는 말을 사용한다. 그렇다면 디버그라는 용어는 어떻게 만들어졌을까?

> 디버그(de-bug) = 버그(벌레)를 집어내다, 에러를 제거하다.

- 앱을 디버깅해서 원인을 찾아 수정하는 일은 지금이라고 크게 다르지않다.
- 코드 조사는 소프트웨어의 문제를 찾아내는 것 뿐 아니라, 작동방식을 이해하거나 기술을 배우기 위한 목적도 있다.

**분석하는 이유는 뭘까?**

- 문제를 찾아낸다.
- 소프트웨어 기능이 어떻게 동작하는지 알아야지, 개선할 수 있다.
- 어떤 기술 또는 라이브러리를 학습한다.

코드의 작동 방식을 탐구하는것은 재미가 있어서 많은 개발자고 좋아한다.<br>
작동 원리를 완전히 이해하게 될 때 희열은 바꿀 수 없다.

- 많은 개발자들은 디버깅 도구를 사용하는 일과, 디버깅을 동일시하는 착각을 많이 하는 편이다.
- 디버거는 개발자가 원하는 커맨드에서 실행을 잠깐씩 멈춰가며 단계별로 코드를 실행함으로써 소스코드를 쉽게<br>
이해할 수 있게 해주는 툴이다.

이 책에서는 총 7가지 방식을 사용한다.

1. 코드 읽기
2. 프로파일링
3. 디버깅
4. 스레드 상태 분석
5. 로그분석
6. 메모리 상태분석
7. 모킹 및 스터빙

대충 읽기만 해도 이해 되는 코드도 있지만, 코드가 미로처럼 길을 잃고 헤매는 코드도 있다.
<br> 대부분의 소스코드는 읽기 쉽지않다. 클린코드, 디자인패턴 다 있지만,
<br> 이걸 지키는 개발자는 얼마나 될까? 레거시 앱은 예전에 만들어졌다고 이 룰에서 제외시켜버린다.<br>
우린 그런 코드까지도 조사할 수 있어야한다.

```java
public int m(int f, int g) {
    try {
      int[] far = new int[f];
      far[g] = 1;
      return f;
    } catch(NegativeArraySizeException e) {
      f = -f;
      g = -g;
      return (-m(f, g) == -f) ? -g : -f;
    } catch(IndexOutOfBoundsException e) {
      return (m(g, 0) == 0) ? f : g;
    }
  }
```
- 위의 코드도 읽기 어렵지만, 훨씬 더 어려운 코드가 많다.
- 이럴때 앱을 이해하려면 다양한 기법을 동원해야 하는데, 가령 프로파일러를 이용하면
<br>어떤 코드가 실행되는지 식별함으로써 조사의 출발점을 결정할 수 있다.

> 앱을 아예 실행하지 못할때도 있고, 크러시가 날때도있다.
<br> 이럴경우 신속하게 분석하고 개선해야한다. 이렇게 앱 크래시 직후에 수집한 데이터를 <br>
살펴보는 것을 **사후조사**라고 한다.
