## 로그를 이용하여 조사하기
이번 장에서는 로그를 사용하면 더 쉽게 이해할 수 있는 몇가지 시나리오를 살펴본다.

- 디버거는 앱 실행의 현재 상태에 집중한다.
- 로그는 과거의 특정 기간에 집중한다 라는 차이가 있다.

로그를 사용할 땐 항상 앱의 표준 시간대를 확인해야한다.
시간이 달라지면 몇시간씩 차이가 나는데, 혼동이 생길 수 있다.

### 로그에 기록된 예외 식별

보통 조사는 어디서 시작할지, 로그를 보고 결정한다. 디버거나 프로파일러 같은 도구 및 기법을 적용해서 조라를 진행한다.

예외 스택트레이스가 로그에서 발견되면 앱 기능에 문제가 있다고 볼 수 있다. 예외 메시지를 보면 문제가 발생한 위치에 관한 힌트를 얻게된다. <br>
예를 들어 NullPotintException은 어떤 커맨드가 변수나 애트리뷰트를 읽지못했을때 발생하는 변수다.

- 예외 발생 라인이 항상 근원이 아니다. 더 근본적인 원인에 지나지않을지도 모른다.

예외가 로그에 기록되었다면, 해당 레퍼런스가 먼저 누락된 이유를 자문해볼 필요가있다.

```java
var invoice = getLastIssuedInvoice();

if(client.isOverdue()){
    if(invoice != null){
        invoice.pay();
    }
}

문제를 지엽적으로 해결하려는 마음이 생기면 일시적으로 해결된 것처럼 보일 뿐이다.
예외 문제의 근본원인이 아닐 수 있다는 생각을 늘 가져야한다.
```

### 예외 스택 트레이스로 어디서 메서드 호출했는지 식별
이럴때는 로깅을 하는게 맞다. 예외 스택 트레이스라고 부르기도 하는데, 코드에서 다음과 같이 예외를 사용하면 충분하다.
```java
new Exception().printStackTrace();
```

### 멀티스레드 아키텍처에서 커맨드의 실행 시간 측정
문제를 일으키는 인숫값을 찾아내려면 쿼리와 그 실행 시간을 로그에 기록하면 된다.<br>
찾아내면 DB 테이블에 인덱스를 하나 더 추가하거나, 튜닝하든지 방안을 강구할 수 있을것이다.

> 예제 코드가 있지만 적지 않겠다. P91

커맨드 실행 시간 측정은 간단하지만 효과적이다. 임시로 잠깐만 사용한다. 이런것은 보통 코드를 오래 보관하면 안된다.

### 멀티스레드 아키텍처에서 커맨드 실행 문제 조사
스레드를 여러개 사용하여 구현하는 멀티 스레드 아키텍처 기반의 앱은 외부 간섭에 민감하게 반응하는편이다.<br>
로그 메시지는 타임스탬프가 포함되어 있다. 그래서 로그 메시지를 타임스탬프 순서로 정렬하면 정확히 흐름 파악 할 수 있다.<br>

```java
String threadName=Thread.currentThread().getName();
```

- 자바 앱은 모든 스레드가 각자 이름을 가지고 있다. 개발자가 부여하거나 JVM이 Thread-x형식으로 명명한다. 
<br>최초는 스레드-0, 스레드-1 이런식으로 가지고 간다.
